# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working
with code in this repository.

Tailor Author - a configurable headless CMS: what content exists is driven
by config/schemas (not hardcoded), extended via a plugin system. pnpm
monorepo, ESM, Node 26+, PostgreSQL (optional Redis). Deep docs in `docs/`.

## Core features (mirrored by `tests/specs/functional/`)

Auth (local/OIDC/invites); a repository **catalog**; the authoring **editor**
(activities, containers, elements, comments, revisions, publish diff);
repository management (structure, settings, workflow, transfer, members);
**publishing**; an **asset** library; cross-repo **content-linking** (live
linked copies); multi-entity **collections**; **admin** (users, groups) and
role-based **access** control.

## Domain model (`docs/concepts/`, `docs/dev/configuration/`)

- **Repository** - top-level content unit (e.g. Course, Knowledge Base);
  its shape is set by a **Schema** (a config blueprint).
- **Activity** - structural node in a Schema's `structure`; may nest
  (`subLevels`, e.g. Module > Page) or stay flat (collections).
- **Content Element** - leaf content block (rich text, video, MCQ) placed
  inside a Content Container.

Four pluggable extension types (`docs/dev/extensions/`): **Content Element**
(`ce`, an editable block); **Content Container** (`cc`, a special-case leaf
Activity laying out elements (Page, Exam) via `templateId`); **Meta Input**
(`mi`, config-attached inputs for a Repository, Activity, or Content Element);
**Plugin** (`pl`, an app-wide feature like i18n: global UI, data hooks, Pinia
state). Packages `@tailor-cms/{ce,tcc,tp}-*` (edit/client + server) aggregate
in `extensions/*/{client,server}.js`, wired by `extension-installer`; scaffold
via `pnpm ce|cc|mi|pl`.

A Schema composes Activities + Content Containers, plus Meta Inputs,
Relationships (non-structural links, e.g. prerequisites), and cross-repo
reuse (`mapsTo` on the source). Registered in `config/src/index.ts`. NOTE:
every schema in `config/src/schemas/` is only an example - none is canonical.

## Commands (repo root unless noted)

- `pnpm setup:dev` - first-time install + interactive env/DB setup.
- `pnpm dev` - boot all (`scripts/boot.js`): waits for Postgres, frees
  ports, runs backend + frontend + api-client watcher. Needs a DB;
  `PORT`/`REVERSE_PROXY_PORT` must differ (FE served via the proxy). `pnpm
  dc` also starts Docker Compose dev services.
- `pnpm build|lint|seed`, `pnpm db:migrate|db:reset`; backend `type-check`.
  Migrations auto-apply at boot unless `DATABASE_DISABLE_MIGRATIONS_ON_STARTUP`.
- Tests (`tests/`): `pnpm e2e:functional|e2e:visual|e2e:a11y`. Single: `cd
  tests && pnpm playwright test specs/functional/<f>.spec.ts
  --project=chrome-admin`; role projects each need their `setup-*` first
  (auth). Page Objects in `tests/pom/` by area - one class per component;
  extend, don't inline.

## Backend - `apps/backend` (full detail in its README.md)

Vertical-slice by domain (`activity/`, `repository/`, `asset/`), each with
`actions/`, `schemas/`, `models/`, `*.service.ts`. One endpoint = one
`*.action.ts` via `defineAction({ body?, query?, params?, handler })`. Zod is
the single source for validation, types, and OpenAPI (`/api/docs`); schemas
named `Input`/`Filter`/`Params`. Actions stay HTTP-shaped; DB logic lives in
services, which throw domain errors that actions map to `createError`. Zod ->
`apps/backend/openapi.json` (rewritten each boot) -> `@tailor-cms/api-client`,
regenerated by `@hey-api/openapi-ts` into `api.<slice>.<method>()` (don't
hand-edit); shared entity types live in `@tailor-cms/interfaces`.

## Frontend - `apps/frontend/app` (Nuxt 4, Vue 3, Vuetify 4, Pinia)

File-based `pages/`/`layouts/`/`middleware/`/`plugins/`; auto-imported
`composables/` + `stores/` (Pinia). Components grouped by area under
`components/{editor,repository,catalog,admin,common,user}`; content-plugin
registries in `lib/content-plugins/`. Prefer dumb/presentational components,
pushing logic into `composables/` (`useX`) and Pinia stores; decompose large
components into focused subcomponents.

## Code style & quality

- **DRY** - reuse `packages/*` (`utils`, `common`, `core-components`,
  `interfaces`) and existing helpers; never copy-paste logic across apps.
- **Shallow nesting** - guard clauses / early returns over nested if/else
  (no "christmas tree"); lift complex conditions into named predicates.
- **Naming** - `is`/`has` boolean props; verb-first action files
  (`get-source.ts`); namespaced `ns:action` events (e.g. `save:element`).
- **Composability** - dumb/presentational components, logic in
  `composables/` (`useX`) and Pinia stores; split large components into
  directories of single-purpose subcomponents.
- **Open/closed** - extend via the extension system (`ce`/`cc`/`mi`/`pl`)
  and config schemas; add new packages instead of editing core.
- **Readable comments** - `/** */` for multiline function docs, 
  no `//` inside JSDoc); `//` for single-line; no `// --- section ---`
  dividers, split the file instead.

## Tooling & conventions

Match style; ESLint `max-len` 80. Never read `.env` (see `.env.example`).
For current library/API docs (Nuxt 4, Vuetify 4, Sequelize, Zod), prefer
the `context7` MCP server (set up in `.mcp.json`) over model memory or
stale web results.
